# CryptoCore

CryptoCore — система защищённой передачи файлов с end-to-end шифрованием, написанная на Go. Ключевой принцип: **приватные ключи никогда не покидают устройство пользователя**. Сервер хранит только публичные ключи, зашифрованные контейнеры и метаданные. Всё шифрование и расшифровка происходят локально на устройстве клиента.

---

## Как работает система: полная картина

### Роли участников

| Участник | Что делает |
|---|---|
| **Клиент-отправитель** | Шифрует файл локально, загружает контейнер на сервер |
| **Клиент-получатель** | Скачивает контейнер с сервера, расшифровывает локально |
| **Сервер** | Хранит публичные ключи и зашифрованные контейнеры. Не может расшифровать ни один файл |

---

## Криптографические примитивы

Система использует только стандартные, хорошо изученные алгоритмы:

| Алгоритм | Назначение |
|---|---|
| **ECDH P-256** | Обмен ключами — вычисление общего секрета между двумя сторонами |
| **ECDSA P-256** | Цифровая подпись — доказательство авторства контейнера |
| **AES-256-GCM** | Симметричное шифрование файла и ключа |
| **HKDF-SHA256** | Вывод ключа из общего секрета |
| **SHA-256** | Хэш файла для проверки целостности |

---

## Ключи пользователя

При регистрации каждый пользователь получает **две пары ключей**:

```
Пара 1 — ECDH (для шифрования):
  приватный ключ ECDH  →  хранится только локально
  публичный ключ ECDH  →  отправляется на сервер

Пара 2 — ECDSA (для подписи):
  приватный ключ ECDSA →  хранится только локально
  публичный ключ ECDSA →  отправляется на сервер
```

Приватные ключи сохраняются в файле `~/.config/cryptocore/profile.json`. Сервер их никогда не видит.

## Шифрование файла: пошагово

Когда Алиса отправляет файл Бобу, происходит следующее:

### Шаг 1 — Генерация файлового ключа

```
fileKey = random(32 байта)   // одноразовый AES-ключ именно для этого файла
```

Каждый файл шифруется своим уникальным ключом. Это защищает от компрометации одного ключа.

### Шаг 2 — Вычисление хэша файла

```
fileHash = SHA-256(plaintext)
```

Хэш будет использован при расшифровке для проверки того, что файл не был изменён.

### Шаг 3 — Формирование AAD (Additional Authenticated Data)

```json
{
  "sender_id": "alice",
  "recipient_id": "bob",
  "file_name": "document.pdf",
  "mime_type": "application/pdf",
  "size": 102400,
  "version": "v1"
}
```

AAD — это метаданные, которые **не шифруются, но аутентифицируются**. AES-GCM гарантирует, что если метаданные были изменены, расшифровка завершится ошибкой. Это защищает от атаки подмены, когда злоумышленник пытается переадресовать файл другому получателю.

### Шаг 4 — Шифрование файла

```
fileNonce   = random(12 байт)
ciphertext  = AES-256-GCM(key=fileKey, nonce=fileNonce, plaintext, aad=AAD)
```

AES-GCM одновременно шифрует и аутентифицирует данные — на выходе получается шифртекст и встроенный MAC-тег.

### Шаг 5 — Ephemeral ECDH и общий секрет

Здесь начинается самое интересное. Алиса не использует свой постоянный ECDH-ключ, а генерирует **одноразовую (ephemeral) пару ключей**:

```
ephemeralPrivKey, ephemeralPubKey = ECDH.GenerateKey()
```

Затем вычисляет общий секрет с постоянным публичным ECDH-ключом Боба:

```
sharedSecret = ECDH(ephemeralPrivKey, bobPublicKey)
```

Почему одноразовый ключ? Это обеспечивает **Perfect Forward Secrecy**: даже если приватный ключ Алисы когда-либо скомпрометирован, прошлые файлы не могут быть расшифрованы, потому что ephemeral-ключ был уничтожен сразу после отправки.

### Шаг 6 — Вывод wrap-ключа через HKDF

```
salt    = random(32 байта)
wrapKey = HKDF-SHA256(ikm=sharedSecret, salt=salt, info="file-wrap-key", len=32)
```

HKDF преобразует «сырой» общий секрет из ECDH в криптографически сильный ключ. Случайный `salt` гарантирует, что каждый раз получается уникальный `wrapKey`, даже если те же два участника обмениваются файлами снова.

### Шаг 7 — Шифрование файлового ключа (key wrapping)

```
wrappedFileKeyNonce, wrappedFileKey = AES-256-GCM(key=wrapKey, plaintext=fileKey)
```

`fileKey` «оборачивается» в `wrapKey`. Теперь `fileKey` может быть передан кому угодно — без `wrapKey` он бесполезен. А `wrapKey` может восстановить только Боб, у которого есть его приватный ECDH-ключ.

### Шаг 8 — Подпись контейнера

```
canonical = JSON(контейнер без поля signature)
signature = ECDSA.Sign(key=aliceSigningPrivKey, msg=SHA-256(canonical))
```

Алиса подписывает весь контейнер своим приватным ECDSA-ключом. Это позволяет Бобу убедиться, что файл пришёл именно от Алисы и не был изменён в пути.

### Итоговая структура контейнера

```json
{
  "version": "v1",
  "sender_id": "alice",
  "recipient_id": "bob",
  "ephemeral_public_key": "<bytes>",
  "key_derivation_salt": "<bytes>",
  "wrapped_file_key_nonce": "<bytes>",
  "wrapped_file_key": "<bytes>",
  "file_nonce": "<bytes>",
  "ciphertext": "<bytes>",
  "file_hash": "<bytes>",
  "metadata": { "file_name": "...", "mime_type": "...", "size": ... },
  "signature": "<bytes>",
  "signature_algorithm": "ECDSA-P256-SHA256",
  "key_agreement_algorithm": "ECDH-P256",
  "hash_algorithm": "SHA-256",
  "created_at": "..."
}
```

Этот JSON сериализуется и загружается на сервер. Сервер не может расшифровать `ciphertext`, потому что у него нет ни `fileKey`, ни приватного ECDH-ключа Боба.

## Расшифровка файла: пошагово

Когда Боб получает контейнер:

### Шаг 1 — Проверка подписи

```
canonical = JSON(контейнер без поля signature)
valid = ECDSA.Verify(key=alicePublicKey, msg=SHA-256(canonical), sig=signature)
```

Боб запрашивает публичный ECDSA-ключ Алисы с сервера и проверяет подпись. Если подпись неверна — расшифровка прерывается. Это защищает от подмены контейнера.

### Шаг 2 — Восстановление общего секрета

```
sharedSecret = ECDH(bobPrivateKey, ephemeralPublicKey)
```

Боб использует **свой приватный** ECDH-ключ и **одноразовый публичный** ключ Алисы из контейнера. В силу математики ECDH результат будет тот же, что вычислила Алиса на шаге 5.

### Шаг 3 — Восстановление wrap-ключа

```
wrapKey = HKDF-SHA256(ikm=sharedSecret, salt=container.key_derivation_salt, info="file-wrap-key", len=32)
```

### Шаг 4 — Расшифровка файлового ключа

```
fileKey = AES-256-GCM.Decrypt(key=wrapKey, nonce=wrappedFileKeyNonce, ciphertext=wrappedFileKey)
```

### Шаг 5 — Расшифровка файла

```
aad = JSON({sender_id, recipient_id, file_name, mime_type, size, version})
plaintext = AES-256-GCM.Decrypt(key=fileKey, nonce=fileNonce, ciphertext=ciphertext, aad=aad)
```

### Шаг 6 — Проверка целостности

```
if SHA-256(plaintext) != container.file_hash → ошибка
```

Хэш подтверждает, что расшифрованный файл идентичен тому, что зашифровала Алиса.

---

Алиса доказала кто она     →  через ECDSA-подпись контейнера
Файл не изменился в пути   →  через тег GCM + проверку подписи
Только Боб может прочитать →  через ECDH: нужен его приватный ключ
Прошлые файлы защищены     →  через одноразовый ephemeral-ключ
Метаданные не подменить     →  через AAD в GCM
Файл не повреждён           →  через SHA-256 хэш

## Аутентификация (challenge-response)

Чтобы получить session token, клиент проходит следующий протокол без передачи пароля:

```
1. Клиент: POST /auth/challenge  { user_id }
   Сервер:  генерирует случайный nonce, сохраняет его (TTL 2 мин)
            → { nonce }

2. Клиент: signature = ECDSA.Sign(signingPrivateKey, SHA-256(nonce))
           POST /auth/verify  { user_id, signature }
   Сервер:  проверяет подпись через сохранённый публичный ключ пользователя
            → { session_token, expires_at }

3. Клиент: использует session_token в заголовке Authorization: Bearer <token>
```

Сервер никогда не видит приватный ключ. Аутентификация доказывает факт владения ключом.

## HTTP API

| Метод | Путь | Auth | Назначение |
|---|---|---|---|
| POST | `/users` | — | Регистрация: сохранить публичные ключи и username |
| POST | `/auth/challenge` | — | Получить nonce для подписи |
| POST | `/auth/verify` | — | Отправить подпись, получить session token |
| GET | `/users/{id}` | Да | Получить публичные ключи по ID |
| GET | `/users/by-username/{username}` | Да | Найти пользователя по никнейму |
| POST | `/files` | Да | Загрузить зашифрованный контейнер |
| GET | `/files/{id}` | Да | Скачать зашифрованный контейнер |

## База данных

```sql
users         — id (auto), username (unique), encryption_public_key, signing_public_key
file_records  — id (auto), sender_id, recipient_id, storage_key, file_name, mime_type, size, created_at
sessions      — token (pk), user_id, expires_at, created_at
challenges    — user_id (pk), nonce, expires_at
```

Зашифрованные контейнеры хранятся **не в базе**, а на файловой системе (Docker volume). В `file_records.storage_key` — путь к файлу. Это позволяет легко перейти на S3/MinIO, заменив только одну реализацию интерфейса `ContainerStorage`.

---

## Почему сервер не может расшифровать файлы

1. `fileKey` никогда не передаётся на сервер в открытом виде — только обёрнутым в `wrapKey`
2. `wrapKey` можно восстановить только зная приватный ECDH-ключ получателя
3. Приватный ECDH-ключ получателя хранится только локально и никогда не покидает его устройство
4. Даже если сервер скомпрометирован и отдаст контейнер злоумышленнику — без приватного ключа получателя файл остаётся недоступным

## Запуск

### Локально (без базы данных, in-memory)

```bash
go run .                  # сервер на :8080
go run ./cmd/desktop      # desktop-клиент
```

```bash
docker compose up --build -d
```

## Инфраструктура

Миграции базы данных управляются через **goose** и встроены в бинарник сервера. При старте сервер автоматически применяет все pending-миграции.

Конфигурация через переменные окружения:

| Переменная | По умолчанию | Описание |
|---|---|---|
| `DATABASE_URL` | — | DSN для PostgreSQL. Если не задана — используется in-memory |
| `DATA_DIR` | `./data` | Директория для хранения контейнеров |
| `ADDR` | `:8080` | Адрес HTTP-сервера |
